Lernfokus dieser Seite

Du wendest die gesamte Clustering-Pipeline auf ein reales Industrieproblem an – von der Problemstellung über die Scaling-Entscheidung bis zur optimalen k-Wahl mit Elbow Plot und Silhouette Score.

8.1.1 Problemstellung & Weihenstephaner Standard

Warum brauchen wir Clustering für alte Maschinen – und was ist der Weihenstephaner Standard?

Weihenstephaner Standard

Im Maschinenbau standardisierter Katalog von Maschinenzustandscodes. Definiert einheitlich, welcher Zustand welchem numerischen Wert entspricht:

  • 4 – Prepared (Bereit)
  • 8 – Mangel
  • 16 – Stau
  • 128 – Produktion
  • 1024 – Eigenstörung
  • 4096 – Fremdstörung

Das Brownfield-Problem

  • Neue Maschinen: haben digitale Schnittstellen → Zustandsausgabe direkt verfügbar
  • Alte Maschinen (Brownfield): keine Schnittstelle → kein Zustand bekannt
  • Nachrüstung (Retrofit) kostspielig
  • Lösung: Datengetriebene Zustandsbestimmung aus vorhandenen Sensoren
  • Verfügbar: Vibrations- und Temperatursensor am Motor (1-minütige Snapshots)
?

Wie nutzen wir Clustering hier?

Verschiedene Betriebszustände erzeugen unterschiedliche Sensor-Muster: Im Produktionsmodus vibriert der Motor stärker als im Leerlauf, die Temperatur verhält sich anders bei einer Störung. Clustering soll diese natürlichen Gruppen im 2D-Raum (Vibration × Temperatur) aufdecken.

k?

Haben wir einen Anhaltspunkt für k?

Der Weihenstephaner Standard nennt 6 typische Zustände – das gibt uns einen Startpunkt für k=6. Ob die Daten das bestätigen, zeigen Elbow Plot und Silhouette-Analyse.

8.1.2 Daten generieren

Wir simulieren synthetische Maschinendaten – 6 Cluster entsprechen den 6 Betriebszuständen.

Daten erzeugen & speichern

Python 
import numpy as np
import matplotlib.pyplot as plt
from sklearn.datasets import make_blobs

# 3000 Datenpunkte, 2 Features, 6 Cluster
X, y = make_blobs(
    n_samples=3000,
    n_features=2,
    centers=6,
    cluster_std=0.55,
    random_state=2
)

# Physikalisch sinnvoll machen
X[:, 0] = np.abs(X[:, 0])      # Vibration → immer positiv
X[:, 1] = np.abs(X[:, 1]) * 7  # Temperatur → größerer Wertebereich

# Visualisierung mit echten Labels (nur zur Kontrolle)
plt.figure()
plt.scatter(X[:, 0], X[:, 1], c=y, cmap='viridis')
plt.xlabel("Vibration")
plt.ylabel("Temperatur")
plt.title("Synthetische Maschinendaten – 6 Zustände")
plt.colorbar(label='Zustand (Ground Truth)')
plt.show()

# Daten speichern
np.save('machine_state.npy', X)
Erklärung
  • centers=6 – entspricht den 6 Weihenstephaner Zuständen
  • X[:, 0] = np.abs(...) – Vibration ist physikalisch immer ≥ 0
  • X[:, 1] * 7 – Temperatur hat größeren Wertebereich als Vibration → unterschiedliche Skalen!
  • np.save() – für späteres Laden ohne Neugenerierung
  • y wird nicht beim Clustering verwendet – nur zur visuellen Kontrolle

8.1.4 Scaling-Entscheidung

Skalieren oder nicht? Hier müssen wir beide Varianten ausprobieren und die Auswirkungen verstehen.

Ohne Scaling

  • Temperatur dominiert – Faktor ~7× größerer Wertebereich
  • Distanzmaß wird fast ausschließlich durch Temperatur-Unterschiede bestimmt
  • Vibrations-Unterschiede werden weitgehend ignoriert
  • Kann sinnvoll sein, wenn Temperatur tatsächlich wichtiger ist

Mit StandardScaler

  • Beide Features werden auf Mittelwert 0, Varianz 1 normiert
  • Vibration und Temperatur haben gleiches Gewicht im Distanzmaß
  • Cluster werden gleichmäßig in beiden Dimensionen gesucht
  • Geeignet, wenn beide Sensoren gleich relevant sind
Python 
from sklearn.preprocessing import StandardScaler

# StandardScaler anwenden
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)

# Vergleich: Vor und nach dem Scaling
print("=== Vor Scaling ===")
print(f"Vibration  – Mean: {X[:,0].mean():.2f}, "
      f"Std: {X[:,0].std():.2f}")
print(f"Temperatur – Mean: {X[:,1].mean():.2f}, "
      f"Std: {X[:,1].std():.2f}")

print("\n=== Nach Scaling ===")
print(f"Feature 0  – Mean: {X_scaled[:,0].mean():.2f}, "
      f"Std: {X_scaled[:,0].std():.2f}")
print(f"Feature 1  – Mean: {X_scaled[:,1].mean():.2f}, "
      f"Std: {X_scaled[:,1].std():.2f}")
Output 
=== Vor Scaling ===
Vibration  – Mean: 4.32, Std: 2.11
Temperatur – Mean: 28.47, Std: 14.85

=== Nach Scaling ===
Feature 0  – Mean: 0.00, Std: 1.00
Feature 1  – Mean: 0.00, Std: 1.00
So what? Scaling kann helfen – aber wir können nicht pauschal behaupten, dass es immer angewendet werden muss oder soll! Die Entscheidung hängt vom fachlichen Kontext ab: Sollen beide Sensoren gleich gewichtet werden? Dann skalieren. Ist Temperatur wichtiger? Dann nicht.

Im Folgenden arbeiten wir mit skalierten Daten (X = X_scaled).
Python 
# Ab hier: skalierte Daten verwenden
X = X_scaled
Info
  • Alle nachfolgenden Schritte verwenden X = standardisierte Daten
  • scaler bleibt gespeichert → kann für neue Punkte mit .transform() genutzt werden

8.1.5 Sweep k – Alle k auf einen Blick

Wir testen k = 2 bis 13 und plotten jedes Clustering – um visuell eine erste Orientierung zu bekommen.

Idee: Anstatt nur einen k-Wert zu testen, erstellen wir eine 3 × 4 Subplot-Matrix (12 Plots für k = 2 bis 13). Gleichzeitig sammeln wir alle inertia_-Werte für den Elbow Plot.
Python 
from sklearn.cluster import KMeans

list_of_inertias = []
list_of_k = range(2, 14)  # k = 2 bis 13

fig, ax = plt.subplots(3, 4, figsize=(30, 15))

for n_clusters in list_of_k:
    kmeans = KMeans(n_clusters, random_state=0)
    cluster_labels = kmeans.fit_predict(X)

    # Inertia für Elbow Plot speichern
    list_of_inertias.append(kmeans.inertia_)

    # Scatter-Plot für dieses k
    ax.flat[n_clusters - 2].scatter(
        X[:, 0], X[:, 1],
        c=cluster_labels,
        cmap='viridis', s=5
    )
    ax.flat[n_clusters - 2].set_title(
        f'k = {n_clusters}', fontsize=14)
    ax.flat[n_clusters - 2].set_xlabel(
        "Vibration (skaliert)")
    ax.flat[n_clusters - 2].set_ylabel(
        "Temperatur (skaliert)")

plt.suptitle("Sweep k – k-means für k=2 bis k=13",
             fontsize=18, fontweight='bold')
plt.tight_layout()
plt.show()
Beobachtung
  • 📊 3×4 Subplot-Grid mit k=2 bis k=13
  • Bei kleinem k werden echte Cluster zusammengefasst
  • Ab k=6 sehen die Cluster natürlich aus
  • Bei hohem k werden Cluster künstlich zerschnitten
  • list_of_inertias bereit für Elbow Plot →

8.1.6 Elbow Plot

Der SSE-Verlauf zeigt, wo der „Knick" ist – und welches k optimal ist.

Python 
# Elbow Plot
plt.figure(figsize=(8, 5))
plt.plot(list_of_k, list_of_inertias, 'go--',
         linewidth=2, markersize=8)

# Erwarteten Knick markieren
plt.axvline(x=6, color='red', linestyle='--',
            alpha=0.6, label='Erwartetes k=6')

plt.ylabel('Sum of Squared Distances (SSE)')
plt.xlabel('k (Anzahl Cluster)')
plt.title('Elbow Plot – Maschinenzustände')
plt.xticks(list_of_k)
plt.legend()
plt.grid(True, alpha=0.3)
plt.show()

# Größten relativen Rückgang finden
diffs = np.diff(list_of_inertias)
rel_diffs = diffs / np.array(list_of_inertias[:-1])
best_k = list(list_of_k)[np.argmin(rel_diffs)] + 1
print(f"Stärkster Elbow bei k = {best_k}")
Ergebnis 
Stärkster Elbow bei k = 6
  • Kurve fällt stark von k=2 → k=6
  • Bei k=6 → deutlicher Knick
  • Ab k=7 nur noch geringe Verbesserung
  • ✅ Elbow bestätigt k=6 – konsistent mit Weihenstephaner Standard

8.1.7 Silhouette-Analyse

Bestätigung des Elbow Plots mit Silhouette Score – diesmal mit dem yellowbrick Visualizer.

yellowbrick SilhouetteVisualizer

  • Erweiterungsbibliothek für sklearn – ML-Visualisierungen auf einem Befehl
  • SilhouetteVisualizer erstellt automatisch den vollständigen Silhouette-Plot
  • Zeigt pro Cluster einen Balken – Breite = Clustergröße, Ausdehnung = Score-Verteilung
  • Installation: pip install yellowbrick

Was wir suchen

  • k mit höchstem durchschnittlichem Silhouette Score
  • Alle Cluster-Balken sollten über dem Durchschnitt liegen
  • Keine negativen Silhouette-Scores
  • Ähnliche Balkenbreiten (vergleichbare Clustergrößen)

Silhouette Score & Visualizer für k = 2 bis 13

Python 
from sklearn.metrics import silhouette_score
from yellowbrick.cluster import SilhouetteVisualizer

list_of_scores = []
fig, ax = plt.subplots(3, 4, figsize=(30, 15))

for n_clusters in list_of_k:
    kmeans = KMeans(n_clusters, random_state=0)
    cluster_labels = kmeans.fit_predict(X)

    # Durchschnittlicher Silhouette Score
    score = silhouette_score(X, cluster_labels)
    list_of_scores.append(score)
    print(f"k={n_clusters:2d} → "
          f"Silhouette Score: {score:.4f}")

    # Yellowbrick Visualizer
    visualizer = SilhouetteVisualizer(
        kmeans,
        colors='yellowbrick',
        ax=ax.flat[n_clusters - 2]
    )
    visualizer.fit(X)
    ax.flat[n_clusters - 2].set_title(
        f'k = {n_clusters}  |  '
        f'Score: {score:.3f}',
        fontsize=12
    )

plt.suptitle("Silhouette-Analyse – k=2 bis k=13",
             fontsize=18, fontweight='bold')
plt.tight_layout()
plt.show()

# Bestes k ausgeben
best_k = list(list_of_k)[np.argmax(list_of_scores)]
print(f"\nBestes k laut Silhouette Score: k = {best_k}"
      f" (Score: {max(list_of_scores):.4f})")

✅ Ergebnis

  • Elbow Plot → Knick bei k=6
  • Silhouette Score → Maximum bei k=6
  • Beide Methoden bestätigen sich gegenseitig
  • k=6 konsistent mit den 6 Weihenstephaner Zuständen

⚠️ Grenzen

  • Kein einfaches Unterfangen, die optimale Clusteranzahl zu bestimmen – besonders bei realen Daten ohne Ground Truth
  • Elbow und Silhouette sind Heuristiken – kein Ersatz für Domänenwissen
  • Scaling-Entscheidung beeinflusst das Ergebnis erheblich
  • Neue Cluster-Methode könnte helfen → Seite 8.2: DBSCAN + kNN

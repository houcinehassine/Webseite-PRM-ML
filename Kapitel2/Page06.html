Model Validation: Grundlagen & Holdout Sets

Sind wir nach dem Training wirklich fertig? Nein – erst die Validierung zeigt, ob das Modell tatsächlich generalisiert. Diese Seite erklärt, warum naiver Ansatz scheitert, wie accuracy_score funktioniert, und wie Holdout Sets das Problem lösen.

Warum validieren? Not-So-Right Way accuracy_score model.score() train_test_split Holdout Sets Generalisierung

Lernfokus dieser Seite

Du verstehst, warum man Validierung braucht (Modell + Hyperparameter müssen Daten repräsentieren). Du kennst das Problem des naiven Ansatzes (Training = Test) und kannst train_test_split korrekt einsetzen, um ein ehrliches Bild der Modell-Performance zu bekommen.

6.1 Warum validieren?

Wir kennen den 4-Schritte-Prozess – aber Schritte 1 und 2 (Modell & Hyperparameter wählen) sind entscheidend. Wie trifft man diese Wahl objektiv?

Der typische ML-Ablauf – Recap

  • ① Modell wählen & entsprechende sklearn-Klasse importieren
  • ② Hyperparameter wählen → Instanziierung
  • ③ Modell trainieren: .fit(X, y)
  • ④ Modell anwenden: .predict(X_new)
  • Die Punkte ① und ② sind von entscheidender Bedeutung

Warum Validierung notwendig ist

  • Um eine objektive Wahl für Modell und Hyperparameter treffen zu können, muss das Modell validiert werden
  • Das Modell und die gewählten Hyperparameter müssen die zugrundeliegenden Daten gut repräsentieren
  • Ohne Validierung: reine Intuition oder Zufall – keine objektive Aussage möglich
  • Ziel: ein Modell, das auf neuen, ungesehenen Daten gut performt (= Generalisierung)

6.2 The "Not-So-Right" Way

Der intuitive Ansatz: trainieren, dann auf einem Teil der gleichen Daten testen. Klingt sinnvoll – ist aber fundamental falsch.

❌ Not-So-Right: Train & Test auf denselben Daten
Training
Test ← dieselben Daten!
Training Set (alle Daten)
Test Set = dieselben Daten → PROBLEM

Schritt 1 — Iris-Datensatz laden

Der Iris-Datensatz (150 Blumen, 4 Features, 3 Klassen) ist ein klassischer sklearn-Beispieldatensatz. Wir laden ihn direkt aus sklearn.

Python – Daten laden 
# Iris-Datensatz laden
from sklearn.datasets import load_iris
iris = load_iris()
X = iris.data    # Feature-Matrix: (150, 4)
y = iris.target  # Labels: (150,)

print("X.shape:", X.shape)
print("y.shape:", y.shape)
print("Klassen:", iris.target_names)
Ausgabe
X.shape: (150, 4)
y.shape: (150,)
Klassen: ['setosa' 'versicolor' 'virginica']
  • 150 Blumen-Messungen
  • 4 Features pro Messung (Kelch/Blütenblatt Länge+Breite)
  • 3 Klassen: Setosa, Versicolor, Virginica

Schritt 2 — Modell trainieren & auf denselben Daten testen

Intuitiver Ansatz: Modell fitten, dann direkt auf denselben Trainingsdaten vorhersagen und die Genauigkeit messen. Das Ergebnis sieht gut aus – aber es ist fundamental falsch.

Python – Not-So-Right Way 
from sklearn.neighbors import KNeighborsClassifier
from sklearn.metrics import accuracy_score

# Modell mit n_neighbors=1
model = KNeighborsClassifier(n_neighbors=1)

# Training auf ALLEN Daten
model.fit(X, y)

# Predict auf DENSELBEN Daten → Problem!
y_model = model.predict(X)

# Accuracy auf Trainingsdaten
accuracy_score(y, y_model)
Ausgabe & Problem
1.0
❌ FAIL
  • Mit n_neighbors=1: jeder Punkt kennt sich selbst als nächsten Nachbarn
  • → 100% Accuracy auf Trainingsdaten
  • Problem: wir trainieren UND validieren auf denselben Daten!
  • Das sagt uns nichts über die echte Performance
❌ Das Problem – Warum FAIL?

Nachdem wir das Modell mit den ausgesuchten Hyperparametern trainiert haben, wenden wir es auf einen Teil dieser selben Daten an und vergleichen die vorhergesagten Werte mit den tatsächlichen. Mit n_neighbors=1 erhalten wir ein Modell, das zu 100% genau die Labels der Trainingsdaten vorhersagen kann – aber das ist kein gutes Zeichen! Das Modell hat die Daten auswendig gelernt. Wir trainieren UND validieren das Modell an denselben Daten – das sagt uns gar nichts über die echte Generalisierungsfähigkeit.

6.3 Accuracy & Model Score – Einschub

Bevor wir den richtigen Weg zeigen: zwei äquivalente Methoden, um die Klassifikationsgenauigkeit zu berechnen – accuracy_score() und model.score().

Methode 1 – Funktion aus sklearn.metrics
accuracy_score(y_test, y_predicted)
  • Importieren: from sklearn.metrics import accuracy_score
  • Input: tatsächliche Labels + vorhergesagte Labels
  • Berechnet: Anteil korrekter Vorhersagen
0.96
Methode 2 – .score() Methode des Modells
model.score(X_test, y_test)
  • In den meisten sklearn-Modellen verfügbar
  • Intern: .predict(X_test) → accuracy_score()
  • Praktisch: kompakter, kein separater Import nötig
0.96

Code — Beide Varianten im Vergleich

Beide Methoden sind äquivalent und liefern dasselbe Ergebnis. Voraussetzung: Modell wurde bereits auf X_train trainiert und wir evaluieren auf X_test (einem getrennten Datensatz).

Python – accuracy_score vs. model.score 
from sklearn.metrics import accuracy_score

# Modell auf Training-Set trainieren
model.fit(X_train, y_train)

# ── Variante 1: accuracy_score() ────────────────
y_predicted = model.predict(X_test)
score_v1 = accuracy_score(y_test, y_predicted)
print("accuracy_score:", score_v1)

# ── Variante 2: model.score() ───────────────────
score_v2 = model.score(X_test, y_test)
print("model.score:   ", score_v2)
Ausgabe
accuracy_score: 0.96
model.score: 0.96
  • Beide Varianten → 0.96 (96% Genauigkeit)
  • accuracy_score(y_test, y_predicted) – explizit, flexibel
  • model.score(X_test, y_test) – kompakt, weniger Code
  • Faustregel: .score() für schnelle Checks,
    accuracy_score() wenn weitere Metriken folgen

Vergleich: beide Methoden

Aspekt accuracy_score(y_test, y_pred) model.score(X_test, y_test)
Import from sklearn.metrics import accuracy_score kein extra Import nötig
Intern vergleicht direkt Labels ruft intern .predict() + accuracy_score() auf
Flexibilität höher – eignet sich für andere Metriken kompakter für Quick-Checks
Ergebnis 0.96 0.96
💡 So what?

accuracy_score(ytest, ypredicted) und model.score(Xtest, ytest) sind äquivalent. Beide berechnen den Anteil korrekt klassifizierter Datenpunkte. Wichtig: X_test und y_test dürfen NICHT im Training verwendet worden sein!

6.4 Holdout Sets – The Right Way

Lösung: Den Datensatz aufteilen in Training- und Test-Set. Training auf dem einen, Evaluation auf dem anderen – das Modell hat das Test-Set nie gesehen.

✅ The Right Way: Train/Test aufteilen
All Data (150 Samples)
Training Set (75 Samples) – .fit(X_train, y_train)
Test Set (75 Samples) – .score(X_test, y_test)
Training Set – Modell wird darauf trainiert
Test Set – zur Evaluation, nie beim Training gesehen

Schritt 1 — Datensatz-Dimensionen prüfen

Bevor wir aufteilen: prüfen wir die Dimensionen unserer Daten. X.shape zeigt (n_samples, n_features), y.shape zeigt (n_samples,).

Python – Datensatz-Dimensionen 
print("X.shape:", X.shape)
print("y.shape:", y.shape)
Ausgabe
X.shape: (150, 4)
y.shape: (150,)
  • 150 Beobachtungen, 4 Features
  • 150 Label-Werte (1D)
  • Genug Daten zum Aufteilen (z.B. 50/50 oder 80/20)

Schritt 2 — train_test_split() – Datensatz aufteilen

sklearn stellt eine vorfertige Funktion bereit: train_test_split(X, y, train_size=<ratio>). Sie mischt die Daten zufällig und teilt sie im gewünschten Verhältnis auf.

Python – train_test_split 
from sklearn.model_selection import train_test_split

# Datensatz 50/50 aufteilen
X_train, X_test, y_train, y_test = train_test_split(
    X, y, train_size=0.5)

# Dimensionen der Splits prüfen
print("X_train:", X_train.shape)
print("X_test: ", X_test.shape)
print("y_train:", y_train.shape)
print("y_test: ", y_test.shape)
Ausgabe
X_train: (75, 4)
X_test: (75, 4)
y_train: (75,)
y_test: (75,)
  • train_size=0.5 → 50% Training, 50% Test
  • 75 + 75 = 150 Samples total ✓
  • Daten werden automatisch gemischt (zufällig)
  • Verwende random_state=42 für Reproduzierbarkeit

Schritt 3 — Training auf X_train + Evaluation auf X_test

Jetzt trainieren wir nur auf X_train / y_train und evaluieren auf dem komplett getrennten X_test / y_test. Das Modell hat das Test-Set nie gesehen – ein ehrliches Bild.

Python – Training & Evaluation (The Right Way) 
from sklearn.neighbors import KNeighborsClassifier

model = KNeighborsClassifier(n_neighbors=1)

# Training NUR auf X_train
model.fit(X_train, y_train)

# Evaluation auf X_test (nie beim Training gesehen!)
score = model.score(X_test, y_test)
print("Test-Accuracy:", score)
Ausgabe
Test-Accuracy: 0.96
✅ Richtig
  • 96% Genauigkeit auf ungesehenen Daten
  • Das ist ein ehrliches Ergebnis
  • Modell kann gut generalisieren

Holdout Sets – Das Prinzip

  • Um die Model Performance nicht an Daten zu testen, die das Modell schon „gesehen" hat → Holdout Sets
  • Datensatz wird in Training- und Test-Sets aufgeteilt
  • Am Training-Set: Modell trainieren/fitten
  • Am Test-Set: Genauigkeit des trainierten Modells überprüfen
  • → Ein genaues Modell kann gut generalisieren

train_test_split() – Parameter

  • X, y – Feature-Matrix und Label-Vektor
  • train_size=0.5 – 50% Training (oder z.B. 0.8 für 80%)
  • test_size=0.2 – alternativ Test-Anteil angeben
  • random_state=42 – für Reproduzierbarkeit (gleiche Aufteilung bei jedem Aufruf)
  • Rückgabe: X_train, X_test, y_train, y_test
💡 So what? – Nachteil von Holdout Sets

Holdout Sets lösen das Problem des naiven Ansatzes – aber sie haben einen Nachteil: Man „verliert" Daten, auf denen man das Modell hätte trainieren können. Bei kleineren Datensätzen ist das problematisch. Außerdem: das Ergebnis hängt von der zufälligen Aufteilung ab. → Lösung: Cross-Validation (→ Seite 7)

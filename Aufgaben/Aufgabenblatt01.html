📚 1 · Grundlagen & Feature-Definition

Worum geht es im Feature Engineering – und was macht ein gutes Feature aus? (Seiten 1–2)

Konzept F1 · Warum ist Feature Engineering so entscheidend für die Modellgüte?
Ein ML-Modell nutzt Features, um Labels vorherzusagen – die Güte des Modells hängt maßgeblich von der Qualität der Features ab, unabhängig vom gewählten Algorithmus.
  • Der größte Aufwand im Data-Science-Prozess steckt in Datenbereinigung und Feature Engineering; die Modellierung macht gefühlt nur ~20 % aus.
  • „Garbage in, garbage out": Auch das beste Modell kann aus schlechten Features keine guten Vorhersagen machen.
  • Feature Engineering ist keine Vorstufe, sondern ein gleichwertiger Teil des ML-Projekts.

Konzept F3 · Definiere präzise, was ein „Feature" ist.
Ein Feature ist eine individuelle, messbare Eigenschaft eines beobachteten Phänomens. Formal: Feature Engineering bildet eine messbare Eigenschaft auf einen Skalar ab.
  • Features können die Daten selbst sein oder daraus abgeleitete Größen (meist aggregiert).
  • Durch das Ableiten von Features verringert man in der Regel bereits die Dimensionalität des Problems.
→ Mehr im Skript: Seite 2 · Was ist ein Feature?
Konzept F4 · Welche drei Eigenschaften muss ein gutes Feature erfüllen?
  1. Informativ – trägt Information über das Phänomen, das wir vorhersagen wollen.
  2. Diskriminativ – erklärt Unterschiede/Strukturen in den Daten (nicht für alle Klassen gleich).
  3. Unabhängig – korreliert nicht zu stark mit anderen Features (vermeidet Redundanz).
Merksatz: Gute Features sind informativ, diskriminativ und (weitgehend) unabhängig. Fehlt eine Eigenschaft, leidet die Modellgüte. → Mehr im Skript: Seite 2 · Drei Eigenschaften
Formel F5 · Was bedeuten N und K in der Feature-Matrix N×K?
N = Anzahl der Samples (Beobachtungen) → jede Zeile ist ein Datenpunkt (z. B. eine Messung, ein Bild, ein Dokument).
K = Anzahl der Features → jede Spalte ist ein Feature (z. B. Temperatur, Frequenz, Pixel-Intensität, Worthäufigkeit).
Der Eintrag xik ist der Wert von Feature k für Sample i. → Mehr im Skript: Seite 2 · Feature-Matrix N×K
Konzept F6 · Warum ist Feature Engineering ein iterativer Feedback-Loop?
Features und Modell beeinflussen sich gegenseitig: Man erzeugt Features, trainiert das Modell, bewertet die Güte und verbessert dann die Features – ein Kreislauf. Im Erzeugen guter Features steckt Domänenwissen: Wir unterstützen das Modell durch unser Verständnis der Daten. Die Pipeline: Quellen → Rohdaten → (Bereinigen + Transformieren) → Features → Modell → Erkenntnisse. → Mehr im Skript: Seite 2 · Iterativer Feedback-Loop

📐 2 · Feature-Raum & Visualisierung

Die Feature-Matrix als geometrischer Raum – und wie t-SNE hochdimensionale Daten sichtbar macht. (Seite 3)

Konzept F7 · Inwiefern ist eine Feature-Matrix mehr als nur eine Tabelle?
Jede Zeile wird zu einem Punkt (Vektor) im K-dimensionalen Feature-Raum – die K Feature-Werte sind die Koordinaten dieses Punktes.
  • Im Feature-Raum werden Cluster, Abstände und Strukturen sichtbar, die in der rohen Tabelle unsichtbar sind.
  • ML-Algorithmen operieren in diesem Raum: k-Means findet Cluster, SVM trennt Klassen durch Hyperebenen.
  • Kernidee: Feature Engineering = den Raum so formen, dass ähnliche Punkte nah beieinander und Klassen trennbar werden.
→ Mehr im Skript: Seite 3 · Geometrische Interpretation
Konzept F8 · Was macht t-SNE, und welche Struktur bleibt dabei erhalten?
t-SNE (t-distributed Stochastic Neighbor Embedding) ist eine nicht-lineare Methode, die hochdimensionale Daten auf 2D/3D projiziert, ohne die lokale Nachbarschaftsstruktur zu zerstören.
  • Im Hochdim. wird je Punktepaar eine Ähnlichkeitswahrscheinlichkeit berechnet (Gauß-Verteilung der Abstände).
  • Im Zielraum werden Punkte iterativ so verschoben, dass diese Ähnlichkeitsstruktur erhalten bleibt (t-Verteilung).
  • Lokale Struktur ≫ globale: nahe Nachbarn bleiben nah; globale Abstände zwischen Clustern sind weniger aussagekräftig.
→ Mehr im Skript: Seite 3 · t-SNE Erklärung
Falle F9 · Der Digits-Datensatz hat 8×8-Pixel-Bilder. Wie viele Features sind das – und warum braucht man t-SNE?
8×8 = 64 Features pro Bild. Ein 64-dimensionaler Raum ist nicht direkt visualisierbar (Menschen sehen max. 3D). t-SNE projiziert die 64 Dimensionen auf 2D, sodass die Ziffern als getrennte Cluster sichtbar werden – obwohl geometrische Konzepte (Abstände, Cluster, Hyperebenen) auch in 64D weiter gelten. → Mehr im Skript: Seite 3 · Embedding Digits

⏱️ 3 · Feature Extraction aus Zeitserien

Vom Rohsignal zum Merkmalsvektor – Strategien zwischen Domänen- und Datengetrieben. (Seite 4)

Formel F10 · Wie lautet die formale Abbildung der Feature Extraction aus einer Zeitserie?
f : ℝT → ℝK – aus einem Rohsignal mit T Messpunkten wird ein kompakter Feature-Vektor mit K Skalaren erzeugt (üblicherweise K ≪ T). → Mehr im Skript: Seite 4 · Mathematische Formulierung
Konzept F11 · Beschreibe das Spektrum „domänengetrieben ↔ datengetrieben".
Feature Extraction ist ein Spektrum zwischen zwei Polen:
  • Domänengetrieben (links): statistische Features (Mittelwert, Varianz, Min, Max, RMS) und Frequenz-Features (FFT, Spektralleistung) – erfordern Domänenwissen, sind aber interpretierbar.
  • Hybrid: Bibliotheken wie tsfresh/TSFEL berechnen hunderte Features automatisch.
  • Datengetrieben (rechts): Autoencoder bzw. Deep Learning (CNN/RNN) lernen Features direkt aus den Rohdaten – flexibel, aber weniger interpretierbar.
Faustregel: Je mehr Domänenwissen, desto weiter links kann man ansetzen. → Mehr im Skript: Seite 4 · Domänen- vs. Datengetrieben
Konzept F12 · Worin unterscheiden sich event-triggered und kontinuierliche Zeitserien?
Event-triggered: Die Messung wird durch ein Ereignis ausgelöst (z. B. Motorstart). Alle Zeitserien haben dieselbe feste Länge T und sind am Trigger ausgerichtet → direkt vergleichbar.
Kontinuierlich: Die Messung läuft ununterbrochen, ohne klares Start-/Endereignis. Für die Extraction schiebt man Fenster (Windows) über das Signal. → Mehr im Skript: Seite 4 · Event vs. Kontinuierlich
Anwendung F13 · Beschreibe das Motorausfall-Szenario als Feature-Extraction-Beispiel.
Ziel: einen Motorausfall erkennen, bevor er passiert. Verfügbare Daten: Stromverbrauch-Zeitserien (Ampere über Zeit). Aus dem Rohsignal (T Messpunkte) werden K Features extrahiert, die einen Ausfall anzeigen (z. B. steigende Grundfrequenz, veränderte Harmonische, zunehmende Varianz). Diese Features gehen dann an ein Klassifikationsmodell (z. B. Random Forest). → Mehr im Skript: Seite 4 · Zeitserien-Szenario

🔤 4 · Encoding & Text Features

Kategoriale Variablen und Texte in Zahlen übersetzen – und die fit/transform-Regel. (Seite 5)

Falle F14 · Warum ist es falsch, Kategorien einfach als 1, 2, 3 … zu nummerieren?
Weil das Modell die Zahlen als Reihenfolge/Größenordnung interpretiert. Bei Apple=1, Banana=2, Coffee=3 schließt das Modell: „Coffee ist dreimal so viel wie Apple" – eine künstliche Ordnung, die inhaltlich nicht existiert und das Modell verzerrt. ML-Modelle sind Rechenmaschinen und brauchen Zahlen, aber die Kodierung darf keine falsche Struktur einführen. → Mehr im Skript: Seite 5 · Kategoriale Variablen
Code F15 · Was macht der LabelEncoder – und wann ist er trotz seines Problems OK?
Der LabelEncoder weist jeder Kategorie eine eindeutige ganze Zahl zu – alphabetisch sortiert (apple=0, banana=1, coffee=2, milk=3). Problem: erzeugt eine künstliche Ordnung.
  • OK ist er nur bei ordinalen Variablen, die wirklich eine Reihenfolge haben (klein=1, mittel=2, groß=3).
  • Code: LabelEncoder().fit_transform(categories) → array([0, 0, 3, 1, 2]).
→ Mehr im Skript: Seite 5 · Label-Encoding
Code F16 · Wie löst One-Hot-Encoding das Ordnungs-Problem – und was ist sein Nachteil?
One-Hot-Encoding gibt jeder Kategorie eine eigene binäre Spalte (0/1). Dadurch sind alle Kategorien gleichwertig – keine künstliche Ordnung.
Nachteil: Aus 1 Spalte werden K Spalten (K = Anzahl Kategorien) → die Dimension steigt stark. Bei sehr vielen Kategorien (z. B. 1000 Produkte) nutzt man stattdessen Feature Hashing. → Mehr im Skript: Seite 5 · One-Hot-Encoding
Falle F17 · Was ist der Unterschied zwischen .fit(), .transform() und .fit_transform() – und wann ist .fit_transform() gefährlich?
.fit() lernt die Parameter (z. B. die Kategorien, Mittelwert, Min/Max). .transform() wendet sie an. .fit_transform() macht beides in einem Schritt.
  • Gefährlich: .fit_transform() auf den Testdaten aufrufen → der Encoder/Scaler lernt aus den Testdaten = Data Leakage.
  • Korrekt: fit_transform nur auf Trainingsdaten, danach auf Testdaten nur transform.
→ Mehr im Skript: Seite 5 · sklearn .fit() / .transform()
Konzept F18 · Was macht Bag-of-Words aus einem Text?
Bag-of-Words zählt die Häufigkeit jedes Wortes in einem Text und repräsentiert das Dokument als Vektor dieser Worthäufigkeiten – ohne die Reihenfolge der Wörter zu beachten. In sklearn: CountVectorizer. Jede Spalte steht für ein Wort des Vokabulars, jeder Wert für dessen Anzahl. → Mehr im Skript: Seite 5 · Bag-of-Words

🧠 5 · Autoencoder & Haar Features

Zwei sehr unterschiedliche Wege, Features zu gewinnen – gelernt vs. geometrisch berechnet. (Seite 6)

Konzept F19 · Beschreibe die Autoencoder-Architektur. Welcher Teil liefert die Features?
Ein Autoencoder besteht aus Encoder → Bottleneck → Decoder:
  • Encoder komprimiert den Input Schicht für Schicht (wichtige Info behalten, Redundanz verwerfen).
  • Bottleneck (Latent Space) – die schmalste Schicht – ist der Feature-Vektor (K Neuronen = K Features).
  • Decoder rekonstruiert den Input; er wird nur fürs Training gebraucht. Für die Feature Extraction nutzt man danach nur den Encoder.
→ Mehr im Skript: Seite 6 · Autoencoder-Architektur
Konzept F20 · Worauf wird ein Autoencoder trainiert, und warum ist er „unüberwacht"?
Er wird darauf trainiert, seinen Input möglichst genau zu rekonstruieren; als Verlust dient typischerweise der Mean Squared Error zwischen Input und Output. Da das Ziel der Input selbst ist, braucht man keine Labels → unüberwachtes Lernen. Anwendungen: Dimensionsreduktion (Alternative zu PCA), Anomalie-Erkennung (hoher Rekonstruktionsfehler = Anomalie), Denoising, generatives Modellieren (VAE). → Mehr im Skript: Seite 6 · Training & Anwendungen
Formel F21 · Wie berechnet ein Haar-Feature einen Skalar aus einem Bild?
Ein Haar-Feature berechnet die Differenz der mittleren Pixelintensitäten zwischen einer hellen (weißen) und einer dunklen (schwarzen) rechteckigen Region:
Haar = Σ(weiße Region) − Σ(schwarze Region) → ein einzelner Skalar.
  • Misst Hell-Dunkel-Muster (Kanten, Konturen, Texturen) und ist robust gegen Beleuchtungsschwankungen.
  • Bekannt aus dem Viola-Jones-Algorithmus zur Gesichtserkennung.
  • Mit dem Integral-Image-Trick in O(1) berechenbar – unabhängig von der Fenstergröße.
→ Mehr im Skript: Seite 6 · Haar-Features
Konzept F22 · Wo im Spektrum (Seite 4) ordnet sich der Autoencoder ein?
Der Autoencoder ist ein datengetriebener Ansatz (rechte Seite des Spektrums). Er lernt Features direkt aus den Daten ohne Domänenwissen – das macht ihn sehr flexibel, aber weniger interpretierbar als z. B. statistische Zeitserien-Features. → Mehr im Skript: Seite 6 · Einordnung ins Spektrum

🎯 6 · Feature Selection

Welche Features sollen ins Modell? Curse of Dimensionality, Occam's Razor und drei Methoden. (Seite 7)

Konzept F23 · Nenne die vier Gründe für Feature Selection.
  1. Curse of Dimensionality – der Feature-Raum wächst exponentiell, Daten werden spärlich.
  2. Garbage In / Garbage Out – irrelevante/rauschbehaftete Features fördern Overfitting.
  3. Occam's Razor – einfachere Modelle generalisieren besser und sind interpretierbarer.
  4. Effizienz – weniger Features = kürzeres Training, weniger Speicher, schnellere Inferenz.
→ Mehr im Skript: Seite 7 · Warum überhaupt?
Falle F24 · Was besagt der „Curse of Dimensionality" und welche Algorithmen leiden besonders?
Mit jeder zusätzlichen Dimension wächst der Raum exponentiell (für gleiche Datendichte: 1 Feature ~10 Punkte, 2 → 100, 3 → 1.000, K → 10K). Dieselben N Punkte werden immer spärlicher verteilt.
  • Abstandsbasierte Algorithmen (k-NN, SVM, k-Means) leiden besonders: in hohen Dimensionen werden alle Abstände fast gleich groß → „nächster Nachbar" wird bedeutungslos.
→ Mehr im Skript: Seite 7 · Curse of Dimensionality
Code F25 · Was macht VarianceThreshold – und berücksichtigt es das Label?
VarianceThreshold entfernt Features, deren Varianz unter einem Schwellwert liegt – ohne das Label zu berücksichtigen (unsupervised).
  • Konstante Features (Varianz = 0, threshold=0.0) tragen keine Information → entfernen.
  • Low Variance: fast konstante Features haben wenig Diskriminationskraft.
  • Vorher: Features mit sehr vielen fehlenden Werten behandeln/entfernen.
→ Mehr im Skript: Seite 7 · VarianceThreshold
Konzept F26 · Was besagt Occam's Razor im Kontext von ML?
„Wenn zwei Modelle gleich gut erklären, bevorzuge das einfachere." In ML heißt das: das Modell mit den wenigsten relevanten Features generalisiert besser auf neue Daten und ist weniger anfällig für Overfitting. Ziel der Selection: die minimale Feature-Menge mit maximalem Informationsgehalt über das Label – ohne redundante oder irrelevante Features. → Mehr im Skript: Seite 7 · Occam's Razor
Code F27 · Wie wählt man mit SelectKBest die besten Features aus?
SelectKBest behält die k Features mit dem höchsten Score bezüglich des Labels. Der Score kommt von einer Scoring-Funktion, z. B. mutual_info_classif (Transinformation für Klassifikation). Beispiel aus dem Skript: 64 Features → 20 auswählen. Im Gegensatz zu VarianceThreshold ist dies supervised (nutzt das Label). → Mehr im Skript: Seite 7 · SelectKBest

⚖️ 7 · Feature Scaling & Normalisierung

Gleiche Gewichtung für alle Features – MinMax, Z-Score und Power Transformations. (Seite 8)

Konzept F28 · Warum muss man Features oft skalieren?
Features haben oft sehr unterschiedliche Wertebereiche (z. B. Alter 25 vs. Einkommen 45.000 €). Ohne Skalierung dominiert das großskalige Feature alle Distanz- und Gradientenberechnungen – kleine Features werden praktisch ignoriert. Nach dem Skalieren tragen alle Features gleichberechtigt bei. → Mehr im Skript: Seite 8 · Warum skalieren?
Falle F29 · Welche Algorithmen sind skalensensitiv – und welche nicht?
Skalensensitiv (immer skalieren): k-NN, SVM, k-Means, PCA, Lineare & Logistische Regression, Gradient Descent, Neuronale Netze – alle, die auf Abständen oder Gradienten beruhen.
Nicht skalensensitiv (Skalieren optional): Decision Tree, Random Forest, Gradient Boosting, Naive Bayes – sie treffen Entscheidungen über Schwellwerte pro Feature, unabhängig von der Skala. → Mehr im Skript: Seite 8 · Skalensensitivität
Formel F30 · Wie lautet die MinMax-Formel – und was ist ihr größter Nachteil?
X′ = (X − Xmin) / (Xmax − Xmin) × (max − min) + min
Skaliert auf einen festen Bereich, standardmäßig [0, 1]. Xmin/Xmax stammen aus den Trainingsdaten.
  • Nachteil – Ausreißer: Ein einzelner Extremwert „streckt" die Skala, alle anderen Werte werden in einen engen Bereich gedrückt.
  • Lösung: RobustScaler (skaliert anhand von Quartilen).
→ Mehr im Skript: Seite 8 · MinMax-Transformation
Formel F31 · Wie lautet die Z-Score-Standardisierung, und auf welche Werte transformiert sie?
Z = (X − μ) / σ mit μ = Mittelwert und σ = Standardabweichung der Trainingsdaten. Sie transformiert Features auf Mittelwert 0 und Standardabweichung 1 und ist die robusteste und meistgenutzte Methode (sklearn: StandardScaler). Anders als MinMax ist sie unempfindlicher gegenüber Ausreißern. → Mehr im Skript: Seite 8 · Standardisierung (Z-Score)
Konzept F32 · Wozu dienen Power Transformations (z. B. Box-Cox)?
Power Transformations (z. B. Box-Cox, sklearn: PowerTransformer) machen schiefe Verteilungen normalverteilungsähnlicher. Das hilft Verfahren, die annähernd normalverteilte Features erwarten, und stabilisiert die Varianz. Sie gehen über reines Skalieren hinaus, weil sie die Form der Verteilung verändern. → Mehr im Skript: Seite 8 · Power Transformations
